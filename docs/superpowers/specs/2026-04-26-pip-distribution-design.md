# mymcp 2.0: pip-based distribution and install simplification

**Status:** Approved design, ready for implementation plan
**Target version:** 2.0.0
**Date:** 2026-04-26

## Goal

Replace the current "git clone + run install.sh" deployment model with a
pip-installable package on PyPI, while keeping production systemd-service
deployments fully supported. Cover three usage scenarios:

- **Production**: long-running systemd service on servers
- **Development**: foreground use on laptops/WSL with no system-level setup
- **Public users**: discoverable on PyPI with low onboarding friction

Python 3.11+ on the target machine is an accepted hard prerequisite.

## Non-goals

- Standalone single-binary builds (PyInstaller/shiv). Reconsider in a future
  release if user demand emerges.
- Docker/OCI images. Out of scope for this work.
- Backward compatibility with `MCP_*` environment variables. This is a hard
  break; one-time migration command is provided.
- New tool features. Tool surface is unchanged.

## High-level architecture

The project is restructured into a standard Python package published to PyPI
under the name `mymcp`. A `mymcp` CLI exposes all functionality (server start,
service install/uninstall, token management, legacy migration) as
subcommands. The current bash `deploy/install.sh` is replaced by
`mymcp install-service`, written in Python and shipped inside the package.

User-facing flows:

```
# Quick try (development)
pipx install mymcp
mymcp serve                    # foreground, prints temp admin+rw tokens to stderr

# Production install
pipx install mymcp
sudo mymcp install-service     # writes /etc/mymcp/.env, systemd unit, log dir
sudo systemctl start mymcp

# Upgrade
pipx upgrade mymcp
sudo systemctl restart mymcp

# Migrate from 1.x install at /opt/mymcp
pipx install mymcp
sudo mymcp migrate-from-legacy
```

**Migration of existing files:**

- `main.py` → split into `src/mymcp/server.py` (FastAPI app factory, no
  side-effects on import) and `src/mymcp/cli.py` (argparse, logging
  configuration, signal handling).
- `mcp_server.py`, `config.py`, `auth.py`, `audit.py`, `metrics.py` → move
  under `src/mymcp/`. Imports updated from bare (`import config`) to
  package-qualified (`from mymcp import config`).
- `tools/` → move under `src/mymcp/tools/` with import path updates.
- `deploy/mymcp.service` → becomes `src/mymcp/deploy/templates/mymcp.service.in`,
  read at install time via `importlib.resources`.
- `VERSION` file → deleted; `setuptools-scm` derives version from git tags.
- `tokens.json` at repo root → deleted (was test artifact).

Tests in `tests/` keep their location but their imports change to
`from mymcp.auth import ...`, etc.

## §2 CLI subcommands

Entry point: `[project.scripts] mymcp = "mymcp.cli:main"`. CLI uses stdlib
`argparse` (no new dependency).

### Token model

Four token kinds exist; CLI exposes management for all of them:

| Kind | Cardinality | Storage | Purpose |
|---|---|---|---|
| `admin` | 1 (single value) | `.env` `MYMCP_ADMIN_TOKEN` | calls `/admin/*` API |
| `metrics` | 1 (single value, empty=disabled) | `.env` `MYMCP_METRICS_TOKEN` | reads `/metrics` |
| `ro` | N (list) | `tokens.json` `role="ro"` | read-only tool calls |
| `rw` | N (list) | `tokens.json` `role="rw"` | all tool calls |

### Subcommands

```
mymcp serve [--env-file PATH] [--host HOST] [--port PORT]
            [--log-level LEVEL] [--log-format text|json]
            [--with-metrics-token]

mymcp install-service [--port PORT] [--bind HOST]
                      [--config-dir DIR] [--log-dir DIR]
                      [--service-user root|mymcp]
                      [--enable-metrics | --no-metrics]
                      [--enable-audit | --no-audit]
                      [--install-ripgrep | --skip-ripgrep]
                      [--yes]

mymcp uninstall-service [--purge]

mymcp migrate-from-legacy [--legacy-dir /opt/mymcp] [--dry-run]

mymcp token list
mymcp token add --name NAME --role ro|rw
mymcp token revoke <TOKEN>
mymcp token rotate-admin
mymcp token rotate-metrics
mymcp token disable-metrics

mymcp version          # also: --version flag at top level
mymcp doctor           # diagnostics: python ver, deps, port, ripgrep, systemd, .env
```

### `serve` configuration discovery

In order, first hit wins:

1. `--env-file PATH` flag
2. `MYMCP_ENV_FILE` environment variable
3. `/etc/mymcp/.env` (system install location)
4. `./.env` (development)
5. None of the above → **temp-token mode**: generate one admin and one rw
   token in memory, print both to stderr, bind to `127.0.0.1` by default.
   Tokens are lost on process exit. Metrics token is not generated unless
   `--with-metrics-token` is passed.

### Logging

Logging is configured in `cli.py` based on `--log-level` (default INFO) and
`--log-format` (default `text`). Library modules only do
`logger = logging.getLogger(__name__)` and never call `basicConfig`. JSON
format uses `python-json-logger` (added to dependencies) for structured logs
suitable for Loki/ELK ingestion.

## §4 `mymcp install-service` flow

Replaces `deploy/install.sh`. Requires root.

### Pre-checks

1. `os.geteuid() == 0`, else error: "Run with sudo"
2. `shutil.which("mymcp")` resolves to a real path; recorded for systemd `ExecStart`
3. `/run/systemd/system` exists (systemd is the active init)

### Argument resolution

If `--yes` is passed, all defaults apply with no prompts. Otherwise prompt for
each value with the default shown. Argument values override prompts.

Defaults:
- `--port 8765`, `--bind 0.0.0.0`
- `--config-dir /etc/mymcp`, `--log-dir /var/log/mymcp`
- `--service-user root`
- `--enable-metrics` (yes), `--enable-audit` (yes), `--install-ripgrep` (yes)

### Steps

1. **Service user**: if `--service-user mymcp`, create system user via
   `useradd -r -s /usr/sbin/nologin mymcp` (skip if already exists).
2. **Port check**: warn if `--port` is in use (`ss -tlnp`); proceed unless
   port is locally listening on the same bind address (then error).
3. **Directories**:
   - `<config-dir>/` mode 750, owner `<service-user>`
   - `<log-dir>/` mode 750, owner `<service-user>`. The audit logger
     creates files inside this directory at runtime; file ownership is
     inherited from the running process so no per-file chown is needed.
4. **`<config-dir>/.env`** (mode 600, owner `<service-user>`):
   ```
   MYMCP_HOST=<bind>
   MYMCP_PORT=<port>
   MYMCP_ADMIN_TOKEN=<32-byte hex>
   MYMCP_METRICS_TOKEN=<32-byte hex or empty>
   MYMCP_TOKEN_FILE=<config-dir>/tokens.json
   MYMCP_AUDIT_ENABLED=<bool>
   MYMCP_AUDIT_LOG_DIR=<log-dir>
   ```
5. **`<config-dir>/tokens.json`**: empty store `{"tokens": {}}` (mode 600).
6. **systemd unit `/etc/systemd/system/mymcp.service`**: rendered from
   `mymcp.service.in` template. Key fields:
   ```
   [Service]
   Type=simple
   User=<service-user>
   EnvironmentFile=<config-dir>/.env
   ExecStart=<resolved-mymcp-path> serve --env-file <config-dir>/.env
   Restart=on-failure
   RestartSec=5
   ```
7. **logrotate** `/etc/logrotate.d/mymcp` if audit enabled.
8. **ripgrep** if requested: try apt/dnf/pacman; fall back to GitHub release
   tarball for matching architecture.
9. `systemctl daemon-reload && systemctl enable mymcp`
10. Print summary block including the generated admin and metrics tokens
    (one-time display).

### `service-user` semantics

- `root` (default): bash tool can execute any system command. Trust model
  assumes Bearer token holders are trusted.
- `mymcp`: bash tool runs as unprivileged user; many sysadmin commands fail.
  Documented as the higher-isolation option for lower-trust clients.

### `uninstall-service`

1. `systemctl stop mymcp; systemctl disable mymcp`
2. Delete `/etc/systemd/system/mymcp.service`
3. Delete `/etc/logrotate.d/mymcp` if present
4. `systemctl daemon-reload`
5. By default keep `<config-dir>` and `<log-dir>` (preserves tokens and audit
   history). `--purge` deletes them after confirmation.

## §5 Migration and release strategy

### Hard-break decision

`MCP_*` environment variables are renamed to `MYMCP_*` with **no compatibility
shim**. Rationale:

- `MCP` is the generic protocol acronym; collision risk grows as the MCP
  ecosystem matures.
- Compat shims accumulate maintenance debt and obscure the real config schema.
- Migration tool is provided and one-shot.

This forces version bump to **2.0.0**. Existing 1.x deployments must run
`mymcp migrate-from-legacy` or recreate `.env` manually.

### `mymcp migrate-from-legacy`

```
sudo mymcp migrate-from-legacy [--legacy-dir /opt/mymcp] [--dry-run]
```

1. Verify `<legacy-dir>/.env` and `<legacy-dir>/tokens.json` exist
2. Parse legacy `.env`, rewrite keys: `MCP_<X>` → `MYMCP_<X>`. Drop
   `MCP_APP_DIR` (no longer meaningful).
3. Write `/etc/mymcp/.env` (mode 600).
4. Copy `<legacy-dir>/tokens.json` → `/etc/mymcp/tokens.json` (mode 600).
5. Run the `install-service` core (steps 6-9) to write the new systemd unit
   and reload daemon.
6. `systemctl stop mymcp` (in case old unit is running) → daemon-reload →
   `systemctl start mymcp`.
7. Print: "Legacy directory `<legacy-dir>` is no longer used. Remove it
   manually after verifying the new service is healthy."
8. `--dry-run` prints all planned actions without performing any.

The command stays available throughout the 2.0.x lifecycle. Removed in 3.0
or after a clear deprecation cycle.

### Release notes / README upgrade section

README gains a "Upgrading from 1.x to 2.0" section documenting:

- Breaking changes list
- One-line migration: `pipx install mymcp && sudo mymcp migrate-from-legacy`
- Manual fresh-install path
- `MCP_*` → `MYMCP_*` key mapping table

## §6 CI and PyPI publishing

### `.github/workflows/ci.yml`

Triggered on PR and push to non-tag refs:

- **lint** job: `ruff check`, `ruff format --check`, `mypy src/mymcp` on Python 3.12
- **test** job: matrix of Python 3.11 / 3.12 / 3.13, runs
  `pytest tests/ -v --benchmark-disable`
- **bats** job: runs the existing
  `tests/test_install.bats tests/test_upgrade.bats tests/test_upgrade_integration.bats`
  to keep the deprecated `deploy/install.sh` and `deploy/upgrade.sh` paths
  validated through the 2.0.x lifecycle. New Python install logic is
  exercised by pytest integration tests under `tests/integration/`.
- **build** job: `python -m build`, uploads `dist/` artifact for inspection

### `.github/workflows/release.yml`

Triggered on push of tag matching `v*`:

1. `python -m build` → `dist/*.whl` + `dist/*.tar.gz`
2. **Build offline bundle**:
   - `pip download dist/*.whl -d offline-bundle/wheels --python-version 3.11
     --platform manylinux2014_x86_64 --only-binary=:all:` (and aarch64)
   - `scripts/fetch-ripgrep.sh` downloads ripgrep static binaries for both
     architectures
   - `scripts/install-offline.sh` is included in the bundle for users
   - tar into `mymcp-<tag>-offline-bundle.tar.gz`
3. **Publish to PyPI** via `pypa/gh-action-pypi-publish@release/v1` using
   OIDC Trusted Publisher (no secrets in repo).
4. **Create GitHub Release** via `softprops/action-gh-release@v2`, attaching
   `dist/*` and the offline bundle, with auto-generated release notes.

### PyPI Trusted Publisher setup (one-time)

1. First release uploaded manually (or via test.pypi.org dry-run) to claim
   the `mymcp` name.
2. On pypi.org project settings, add Trusted Publisher: GitHub Actions,
   `algony-tony/mymcp` repo, workflow `release.yml`, environment `pypi`.
3. Subsequent releases use OIDC; no API token in the repo.

## §7 Bash tool SIGTERM cleanup

Current `tools/bash.py` uses `subprocess.run`, which makes spawned children
orphans when the server process exits (they get reparented to PID 1 and keep
running).

### Changes

1. Each bash invocation uses `subprocess.Popen(..., start_new_session=True)`,
   creating a new process group. Track the `Popen` object in a thread-safe
   weak-reference set (`weakref.WeakSet`).
2. `cli.py` `serve` command registers SIGTERM and SIGINT handlers that:
   - Iterate the in-flight set, call `os.killpg(p.pid, SIGTERM)` for each
   - Wait up to 5 seconds (configurable via `MYMCP_SHUTDOWN_GRACE_SEC`,
     default 5)
   - For any still alive: `os.killpg(p.pid, SIGKILL)`
   - Then signal uvicorn to begin its own graceful shutdown
3. The existing per-call `timeout` behavior is preserved; cleanup logic is
   additive.

### Test

A new pytest integration test under `tests/integration/test_signal_cleanup.py`:

1. Spawn `mymcp serve` as a subprocess with a temp `.env`
2. Issue a `bash_execute` call running `sleep 30`, capture the child PID
   from process listings
3. Send SIGTERM to the `mymcp serve` PID
4. Within `MYMCP_SHUTDOWN_GRACE_SEC + 2` seconds, assert the `sleep` PID
   is gone (`os.kill(pid, 0)` raises `ProcessLookupError`)

## Configuration reference (after `MCP_*` → `MYMCP_*` rename)

| Variable | Default | Purpose |
|---|---|---|
| `MYMCP_HOST` | `0.0.0.0` | uvicorn bind |
| `MYMCP_PORT` | `8765` | uvicorn port |
| `MYMCP_ADMIN_TOKEN` | (required) | admin Bearer for `/admin/*` |
| `MYMCP_METRICS_TOKEN` | empty | metrics Bearer; empty disables `/metrics` |
| `MYMCP_TOKEN_FILE` | `<config-dir>/tokens.json` | ro/rw token store path |
| `MYMCP_AUDIT_ENABLED` | `false` | enable audit logger |
| `MYMCP_AUDIT_LOG_DIR` | `/var/log/mymcp` | audit log directory |
| `MYMCP_ENV_FILE` | (none) | overrides default `.env` discovery |
| `MYMCP_SHUTDOWN_GRACE_SEC` | `5` | seconds to wait for bash children on SIGTERM |

Protected paths (always inaccessible to file tools): the directory containing
the loaded `.env` file and `MYMCP_AUDIT_LOG_DIR`. The legacy
`MYMCP_APP_DIR` concept is removed.

## Out-of-scope follow-ups

These are explicitly deferred to later versions to keep 2.0 focused:

- Decorator-based tool registration (currently a manual dispatch chain)
- Standalone PyInstaller binaries
- Docker images
- Tool-level rate limiting / quotas

## Acceptance criteria

A user on a clean Ubuntu 24.04 machine with Python 3.11 should be able to:

```
pipx install mymcp
mymcp serve                    # works, prints temp tokens, accepts MCP traffic on 127.0.0.1:8765
sudo mymcp install-service --yes
sudo systemctl start mymcp     # service starts, /metrics gated by token, /health returns 200
sudo mymcp token add --name laptop --role rw   # adds usable rw token
sudo mymcp uninstall-service --purge           # removes everything cleanly
```

A user on an existing 1.x install at `/opt/mymcp` should be able to:

```
pipx install mymcp
sudo mymcp migrate-from-legacy
sudo systemctl status mymcp    # new service is running, old tokens still work
```

A user behind an air gap should be able to download
`mymcp-2.0.0-offline-bundle.tar.gz` from a GitHub Release and run the bundled
`install-offline.sh` to complete the same `install-service` flow without
internet access.
